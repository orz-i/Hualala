# AI 剧集生产协作平台设计文档

## 1. 背景与定位

本项目面向小型短剧工作室与 MCN，目标不是做单点 AI 生成工具，而是做一套覆盖选题策划、世界观设定、分集生成、剧本生成、分镜拆解、视觉资产生成与动态样片预演的生产协作平台。

平台的核心价值是协作管控，而不是单次生成。AI 在系统中扮演半自动流水线节点的角色，负责生成候选版本、提供改写建议和批量拆解能力；所有关键节点仍由人工审核、回退和定稿。

在视觉资产侧，平台不要求素材必须由自身生成。第一阶段默认支持外部图片 / 视频素材导入，并将平台重点放在资产接入、归类、挂接、审核与复用，而不是将平台内生成作为前提能力。

第一年按私有化交付标准设计，后续再将共性能力标准化为 SaaS。

## 2. 目标用户与核心价值

### 2.1 目标客户

- 小型短剧工作室
- MCN 内容团队

### 2.2 核心角色

- 老板 / 制片人
- 主编剧
- 策划
- 运营
- 审核 / 法务
- 管理员 / 运维

### 2.3 付费驱动点

本平台优先解决协作管控问题，包括：

- 项目流转与阶段管控
- 多角色协同
- 审批与留痕
- 版本管理与回退
- 模板与资产复用
- AI 任务编排与结果治理

## 3. 产品边界

### 3.1 第一阶段覆盖链路

第一阶段覆盖两条并行主线：

#### 业务主线

1. 项目立项
2. 世界观与角色设定
3. 分集大纲生成
4. 剧本初稿生成与改写
5. 分镜拆解与镜头包生成
6. 外部视觉资产导入、挂接与筛选
7. 动态样片预演

#### 治理主线

1. 高价值 AI 长链路工作流编排
2. 任务状态、重试、取消与恢复
3. 使用量计量与成本归集
4. 组织级 / 项目级额度、预算、预警与熔断
5. 审计留痕与事件可追踪

### 3.2 产品主轴

产品主轴确定为：

`协作平台优先，可靠工作流编排与成本治理前置，AI 动态样片仍为高阶模块`

这意味着动态样片是平台能力的一部分，但不是第一阶段的唯一价值锚点，也不应反向决定整个平台的一期架构。

### 3.3 非目标

第一阶段不追求以下目标：

- 纯自动无人审核的全流程创作
- 以视频生成能力作为平台主入口
- 一开始就做完整开放式 SaaS 计费体系
- 一开始就拆成重微服务架构
- 一期即完整落地导演意图标准化子系统的独立页面与完整流程
- 假设图片 / 视频资产必须通过平台内能力生成

### 3.4 已识别高风险点与一期处理策略

在当前业务沟通中，已经识别出一个高频且高成本风险点：

- 编剧 / 导演负责文字分镜
- 抽卡执行人员负责图片 / 视频生产
- 团队主要依赖线上沟通交接
- 导演对构图重点和镜头语言的意图在交接中容易失真
- 结果往往在整批镜头完成后才被发现不可用，导致整批返工

该问题会显著影响图片与视频生产成本，但当前判断是不将其上升为一期独立主系统，而是采用以下策略：

1. 整体平台主线保持不变
2. 将该问题记录为高优先级产品风险
3. 一期先预留数据结构与流程边界
4. 待整体系统稳定后，再将其升级为独立能力模块

## 4. 总体架构

### 4.1 前端架构

前端采用双端分离：

#### 创作端

- 技术：Tauri 2.0 + React + Tailwind CSS + shadcn/ui
- 形态：桌面创作工作台
- 面向角色：制片、主编剧、策划、运营
- 核心职责：
  - 项目工作区
  - AI 流程触发
  - 剧情、角色、剧本、分镜编辑
  - 版本对比与回退
  - 资产浏览与引用
  - 本地缓存与断点续作

#### 管理端

- 技术：React + Tailwind CSS + shadcn/ui
- 形态：Web 管理后台
- 面向角色：管理员、审核、老板、运维
- 核心职责：
  - 组织与成员管理
  - 角色与权限配置
  - 模型与提示词治理
  - 审批流配置
  - 成本统计与审计日志
  - 私有化交付配置

### 4.2 后端架构

- 语言：Golang
- 数据库：PostgreSQL
- 媒体存储：对象存储
- 业务协议：Connect RPC
- 事件通道：SSE
- 文件通道：上传会话 + 对象存储直传 / 后端中继

后端建议采用模块化单体或边界清晰的服务化单体设计，核心模块包括：

1. 身份与组织中心
2. 项目与内容中心
3. 镜头执行中心
4. 媒体资产中心
5. 工作流编排中心
6. 任务执行与恢复中心
7. 成本计量与预算守卫中心
8. 审批与审核中心
9. 配置与审计中心

### 4.3 关键架构原则

- 管理端保留 Web，不将全部系统职责压进 Tauri
- AI 能力通过工作流节点挂接，不直接嵌入业务正文
- 高价值长链路任务应接入工作流引擎，普通异步任务可继续使用轻量任务体系
- 所有高成本 AI 动作都必须经过额度、预算与策略守卫
- 模型、对象存储、消息队列等基础能力应可替换
- 组织级配置与项目级内容严格分层
- 视觉资产链路必须按“来源无关”设计，兼容人工上传、外部平台导出、外包交付等多种接入方式

### 4.4 通信架构

前后端通信采用三层设计：

1. `Connect RPC` 负责业务查询与业务命令
2. `SSE` 负责长任务状态与业务事件推送
3. `Upload Session` 负责图片 / 视频等大文件导入

对于未来的多人实时协同编辑，建议单独增加第四类通信通道，但不纳入 Phase 1 主实现：

4. `WebSocket + CRDT` 负责剧本、分镜板等高实时共编场景

#### Connect RPC 适用范围

Connect RPC 负责以下业务接口：

- 认证与当前身份
- 组织、成员、角色与权限
- 项目、集数与阶段信息
- Story Bible、角色、剧本、分镜、镜头等正文对象
- 工作流实例启动、查询、重试与取消
- 导入批次创建、素材挂接确认、主素材切换
- 审批、抽检、打回等流程动作
- 预算快照、使用量明细、成本事件与预算策略管理

#### SSE 适用范围

SSE 仅用于服务端向前端推送状态变化，主要包括：

- `job.updated`
- `workflow.updated`
- `import_batch.updated`
- `shot.updated`
- `approval.updated`
- `preview_cut.updated`
- `budget.updated`
- `billing.alert`
- `system.notice`

SSE 不承担剧本、分镜板等正文对象的多人实时协同编辑。该类场景在后续阶段应采用 `WebSocket + CRDT` 架构，避免多人并发修改时出现版本覆盖与离线合并冲突。

#### Upload Session 适用范围

上传会话负责：

- 创建导入批次后的文件上传协调
- 对象存储直传或后端中继上传
- 上传完成回调
- 文件解析与批量匹配触发

图片 / 视频二进制本体不走 Connect RPC。

- `hualala/auth/v1`
- `hualala/org/v1`
- `hualala/project/v1`
- `hualala/content/v1`
- `hualala/execution/v1`
- `hualala/workflow/v1`
- `hualala/billing/v1`
- `hualala/asset/v1`
- `hualala/review/v1`
- `hualala/common/v1`

一期优先落地的服务包括：

- `AuthService`
- `ProjectService`
- `ContentService`
- `ExecutionService`
- `AssetService`
- `WorkflowService`
- `BillingService`
- `ReviewService`

### 4.6 鉴权、组织上下文与权限

平台鉴权分为三层：

1. 身份认证：确认用户是谁
2. 组织上下文：确认当前在哪个组织中操作
3. 资源权限：确认当前用户在该组织、项目、对象上的可执行动作

一期建议采用：

- Web 管理端：`HttpOnly Cookie + Session / Refresh`
- Tauri 创作端：`Access Token + Refresh Token`
- 请求上下文：通过 header 或 Connect metadata 传递组织上下文
- 权限模型：`RBAC + Project Scope`

Connect RPC、SSE、Upload Session 三层都必须共用同一套组织隔离与权限判断逻辑。

### 4.7 后端模块与数据库边界

一期后端建议按以下模块组织：

- `auth`
- `org`
- `project`
- `content`
- `execution`
- `asset`
- `workflow`
- `usage`
- `billing`
- `review`

数据库主表建议按以下业务边界规划：

- 组织与权限：`organizations`、`users`、`memberships`、`roles`、`role_permissions`
- 项目与内容：`projects`、`episodes`、`story_bibles`、`characters`、`scripts`、`storyboards`、`shots`
- 执行与审核：`shot_executions`、`shot_execution_runs`、`approvals`、`shot_reviews`、`state_transitions`
- 工作流与成本治理：`workflow_runs`、`workflow_steps`、`usage_records`、`budget_policies`、`billing_events`
- 资产接入：`import_batches`、`upload_sessions`、`upload_files`、`media_assets`、`import_batch_items`、`shot_asset_links`、`shot_candidate_assets`、`asset_links`
- 任务与事件：`jobs`、`event_outbox`

## 5. 核心业务对象设计

平台数据模型分为四层。

### 5.1 治理对象

- `Organization / Tenant`
- `User`
- `Membership`
- `Role / Policy`
- `Model Profile`
- `Prompt Policy`
- `Budget Policy`

这层用于组织隔离、角色权限、模型治理与私有化配置。

### 5.2 生产对象

- `Project`
- `Story Bible`
- `Character`
- `Episode`
- `Script`
- `Storyboard`
- `Shot`

这些对象描述内容生产的真实结果，是平台的长期真相。

### 5.3 运行对象

- `Shot Execution`
- `Shot Execution Run`
- `Workflow Definition`
- `Workflow Run`
- `AI Job`
- `Usage Record`
- `Billing Event`
- `Review Task`
- `Approval`
- `Version Snapshot`

这些对象描述生成、审核、驳回、重跑等过程，不应直接混入正文模型。

### 5.4 媒体对象

- `Media Asset`
- `Import Batch`
- `Reference Frame`
- `Audio Track`
- `Track Binding`
- `Shot Pack`
- `Preview Cut`
- `Asset Library`

这层为图片、音频、视频、镜头包和动态样片提供统一承载边界。

### 5.5 一期为导演意图标准化预留的对象边界

虽然一期不单独实现完整的导演意图标准化子系统，但建议提前预留以下对象能力：

- `Shot`：作为分镜下的镜头单元，支持独立状态流转
- `Shot Execution`：作为镜头当前执行态对象，承载当前主素材、责任人、执行状态与提审动作
- `Shot Execution Run`：作为镜头执行轮次对象，承载每次执行、返工、重跑和工作流绑定信息
- `Reference Frame`：作为镜头执行参考图的独立资产类型
- `Provenance Credential`：用于保存素材来源凭证、AI 生成声明和后续 C2PA 兼容元数据
- `Track Binding`：用于表达镜头时间轴与对白、拟音、环境音、配乐等多轨音频的绑定关系
- `Shot Composition Notes`：用于保存构图重点、主体位置、必须出现元素、禁止项等结构化字段
- `Shot Review Reason`：用于记录导演或审核打回原因标签
- `Primary Asset` / `Candidate Assets`：用于表达单镜头的主素材与候选素材集合
- `Import Batch`：用于管理一轮素材导入的来源、批次、自动匹配结果与人工确认结果
- `Shot Size`：用于描述镜头类型，如全景、中景、近景、特写
- `Camera Move`：用于描述运镜方式，如固定、推拉、平移、跟拍
- `Subject Action`：用于描述主体动作的完整语义
- `Continuity Notes`：用于描述服装、光源方向、道具状态等连续性要求

这些能力可以先作为数据结构与状态位存在，不要求一期必须提供完整单独页面。

## 6. 数据建模规则

平台需要坚持以下建模规则：

1. 所有正文内容必须版本化，不能只保留一份可变正文
2. AI 运行结果先落到新快照，而不是直接覆盖生产对象
3. 审批与评论属于过程数据，不属于正文结构
4. 角色、场景、参考图、样片都应独立成资产对象，而不是普通附件
5. 动态样片必须保留输入分镜版本、生成参数、输出媒体之间的关联链
6. `Shot` 只承载结构骨架，不持有主素材、执行责任人和审核结论
7. `Shot Execution` 只表达镜头当前执行态，是执行工作台的唯一当前态真相
8. `Shot Execution Run` 表达镜头历史轮次，是工作流、返工和成本归因的绑定点
9. 审核结论通过 `ShotReview` 事件流表达，不直接回写成 `Shot` 主状态
10. 镜头层级必须允许挂接执行参考图、构图备注和打回原因，为后续导演意图标准化模块预留空间
11. 视觉资产必须允许双挂接，既可挂到镜头，也可挂到角色 / 场景，以兼顾当前执行与后续复用
12. 单镜头素材必须支持“1 份主素材 + 多份候选素材”的结构，但主素材真相应收口到 `Shot Execution`
13. `Shot` 必须承载镜头类型、运镜方式、主体动作和连续性备注，作为后续审核、抽检、模板复用和导演意图标准化的核心字段
14. 媒体资产必须预留来源凭证挂载位，兼容后续 `C2PA Content Credentials` 与人类贡献记录
15. 音频资产必须独立于单一视觉主素材建模，为对白、拟音、环境音、配乐等多轨结构预留绑定空间

## 7. 核心流程设计

一期的核心流程由两条并行主线组成：业务生产主线负责把内容从立项推进到样片预演，治理主线负责保证高价值 AI 长链路可恢复、可追踪、可控成本。两条主线共享项目、镜头、任务和事件边界，但职责不同。

一部剧的业务主流程拆为六段：

### 7.1 项目立项

- 创建项目
- 选择工作流模板
- 指派成员与角色
- 定义交付目标和预算边界

### 7.2 世界观与角色设定

- 维护 Story Bible
- AI 扩写人物设定与关系
- 人工合并、删改和定稿
- 沉淀角色与设定资产

### 7.3 分集与剧本生成

- 按集数批量生成分集与剧本初稿
- 支持局部重写和版本对比
- 主编剧审核并推进状态

### 7.4 分镜与镜头包

- 从剧本拆解分镜卡
- 关联角色、场景、情绪、镜头运动、时长
- 形成可复用的 Shot Pack
- 生成 `Shot` 结构骨架，而不是直接承载执行态
- 为镜头预留构图重点、参考图挂载位和审核标签
- 镜头结构对象需保存 `shot_size`、`camera_move`、`subject_action` 与 `continuity_notes`

### 7.5 视觉资产生成

第一阶段视觉资产能力调整为导入优先，而非生成优先，主流程包括：

- 外部素材手动上传与批量导入
- 以 `Import Batch` 管理导入来源、项目 / 集数、任务批次和归类状态
- 支持按文件名 / 目录规则自动匹配，或通过清单导入批量绑定
- 导入确认后更新 `Shot Execution` 当前态，并生成或推进对应 `Shot Execution Run`
- 支持人工确认挂接结果
- 资产可挂接到镜头执行态，也可挂接到角色 / 场景
- 每个镜头执行态保留 `1 份主素材 + 多份候选素材`
- 素材进入项目资产库与组织资产库，供后续复用与预演消费

### 7.6 动态样片预演

- 基于 Shot Pack、`Shot Execution Run` 与视觉资产生成样片
- 作为评审产物，而非第一阶段最终交付物
- 支持失败重试与降级输出

### 7.7 工作流编排与成本治理

- 对高价值长链路任务建立 `Workflow Run`，而不是仅依赖平面 `Job` 轮询
- 对普通异步任务保留 `jobs` 体系，避免一期一次性重构所有后台动作
- 工作流绑定点应优先落在 `Shot Execution Run` 等执行轮次对象，而不是直接绑定 `Shot`
- 所有高成本任务在启动前都必须经过额度、预算与策略守卫
- 执行过程持续写入使用量记录、预算事件和可观测状态
- 通过 SSE 把工作流状态、预算变化和成本预警推送到双端

## 8. 人工关卡与自动节点

### 8.1 必须人工把关的节点

- 项目立项确认
- Story Bible 定稿
- 分集审核
- 剧本定稿
- 分镜过稿
- 样片确认

对于图片 / 视频生产阶段，一期建议至少保留以下轻量人工约束能力：

- 镜头级参考图挂载
- 打回原因标签化
- 小批量样例抽检
- 导演按批次抽检与关键镜头介入

### 8.2 适合自动串行的节点

- 角色设定扩写
- 人物关系润色
- 分集大纲生成
- 剧本初稿生成
- 分镜拆解
- 静态图批量生成

原则是：自动节点输出候选版本，人工关卡决定是否推进。

## 9. 商业化路线图

### 9.1 Phase 1：MVP 试点版

目标：验证真实工作室流程能否在系统内跑通。

聚焦能力：

- 项目、成员、角色、审批流
- 世界观、角色、分集、剧本、分镜文本链路
- `Shot` 层级模型与镜头状态流转

## 10. 风险与交付保障

### 10.1 AI 质量风险

风险表现：

- 输出偏题
- 风格不一致
- 角色设定冲突
- 分镜不可执行

应对策略：

- 以 Story Bible 作为统一约束源
- 保留版本差异与局部重跑能力
- 关键节点人工审核

### 10.2 成本与算力风险

风险表现：

- 图片与视频生成成本过高
- 试错阶段资源浪费
- 视频任务拥堵

应对策略：

- 组织级和项目级额度控制
- 使用量记录、价格快照与预算事件留痕
- 预算阈值预警与策略熔断
- 队列与优先级调度
- 视频失败时降级到静态预演
- 不将视觉资产平台内生成作为主流程前提，优先兼容外部导入与外部生产

### 10.3 长链路工作流可靠性风险

风险表现：

- 长链路 AI 任务节点多、耗时长、外部依赖多
- 单节点失败后整条链路被迫重跑
- 任务恢复、取消、补偿与排障缺乏统一边界

应对策略：

- 高价值长链路接入工作流引擎
- 普通异步任务保留轻量任务体系
- 工作流执行历史、重试与恢复点独立建模
- 事件推送与任务查询保持统一前端观察面

### 10.4 私有化交付风险

风险表现：

- 企业客户要求替换模型或存储
- 审计与部署要求高
- 后期被迫大规模重构

应对策略：

- 模型网关抽象
- 存储适配层抽象
- 配置中心与审计中心前置设计

### 10.5 协作冲突风险

风险表现：

- 多人并发修改正文
- AI 输出与人工修改冲突
- 审批链断裂

应对策略：

- 对象级状态锁或编辑租约
- 草稿版本与发布版本分离
- 评论、指派、审批全程留痕
- Phase 1 不把 SSE 误用为实时共编协议
- 后续多人正文共编采用 `WebSocket + CRDT`，解决弱网与离线重连下的合并冲突

### 10.6 导演意图交接失真风险

风险表现：

- 文字分镜进入抽卡执行后，对构图重点的理解出现偏差
- 不同执行人对镜头语言和画面重心理解不一致
- 导演往往在整批镜头完成后才发现不可用
- 单集 20+ 镜头被整批打回，导致时间和成本浪费

一期处理策略：

- 在镜头对象上预留参考图挂载能力
- 在镜头对象上预留构图备注与必须元素字段
- 打回意见标签化，而不是只保留聊天记录
- 支持对图片 / 视频产出做小批量抽检，而不是整批完成后统一审片
- 分镜负责人 / 编剧负责素材挂接确认与主素材拍板，导演主要做抽检与关键镜头把关

### 10.7 视觉资产接入复杂度风险

风险表现：

- 图片 / 视频素材来源混杂，来自本地、外部平台或外包团队
- 素材命名不统一，无法直接自动归类
- 同一镜头存在多份候选素材，容易在群聊和网盘中丢失主版本

一期处理策略：

- 引入 `Import Batch` 管理一轮导入
- 支持手动挂接、规则匹配、清单导入三种方式并行
- 自动匹配后必须人工确认
- 单镜头保留主素材与候选素材集合

### 10.8 通信与鉴权复杂度风险

风险表现：

- 双端通信协议不统一，前后端接口容易漂移
- 长任务状态散落在轮询和页面局部状态中，导致前端一致性差
- Upload Session、SSE、业务 API 使用不同鉴权口径，容易出现越权或数据不可见

一期处理策略：

- 业务层统一收敛到 Connect RPC
- 状态推送统一收敛到 SSE
- 大文件传输统一收敛到 Upload Session
- 组织上下文通过统一 metadata / header 进入服务层
- 使用统一的权限检查器处理 Connect、SSE 与上传会话

后续升级方向：

- 将导演意图标准化做成独立模块
- 增加结构化镜头卡
- 增加执行基准图确认流
- 增加镜头一致性质检
- 增加素材来源凭证与 C2PA 兼容导入 / 导出
- 增加多轨音频绑定与时间轴级审阅

## 12. 技术结论

基于当前业务目标，技术选型结论如下：

- 后端：Golang
- 数据库：PostgreSQL
- 业务通信：Connect RPC
- 事件推送：SSE
- 高价值工作流：Temporal
- 文件导入：Upload Session + 对象存储
- 创作端：Tauri 2.0 + React + Tailwind CSS + shadcn/ui
- 管理端：React + Tailwind CSS + shadcn/ui
- 媒体存储：对象存储

这套选型与当前产品边界匹配，没有明显方向性错误。真正需要谨慎的不是框架本身，而是平台是否坚持“协作平台优先、AI 能力模块化接入”的设计纪律。
